## Campaign harness (shipped 2026-05-10)

Two scripts under `scripts/` + npm aliases (`simulate-campaigns`,
`load-test-campaigns`) that together stress-test the Image Creator +
Director-agent stack end-to-end.

- `scripts/simulate-campaigns.mjs` synthesizes a wipeable marketing dataset
  (products / multi-channel campaigns / daily metrics / pre-populated vibe
  boards / cloned media library) under a dedicated harness user.
  All simulated rows are namespaced (`source='simulated'`,
  `tag='simulated-clone'`, `state.meta_source='simulated-board'`) so the
  real admin data + `seed-timeline` outputs are untouched.
- `scripts/run-campaign-load.mjs` drives the Director against the simulated
  boards with `--concurrent` parallelism, captures p50/p95/p99 latencies
  at every stage (session-create / first-event / ttf-agent-msg /
  ttf-first-generation / finished), writes a JSON report to
  `scripts/.out/run-*.json`.

See `scripts/CAMPAIGN-HARNESS.md` for full docs.

**Observed during validation:** 8/8 boards succeeded at concurrency 4 with
Gemini-fast; p50 ttf-msg 6.1s, p95 9.7s; 16 image generations in 144s
wall-clock. The runner exercised 26 harness-dispatched custom tools — see
"observed bottlenecks" below.

**Open follow-ups surfaced by the harness (do NOT block, but worth a pass):**

1. **`/runs/:id` background drain is unreliable in supabase-local.** The
   `runInBackground(...)` in `supabase/functions/runs/index.ts:94` relies on
   `EdgeRuntime.waitUntil`. When that's missing, the floating promise gets
   dropped between requests, so `refreshFromAnthropic` + `drainImageToolCalls`
   never materialize events or post tool_results. Effect: the agent stalls
   at the first `read_board`. **The harness works around this** by polling
   Anthropic's `/v1/sessions/:id/events` directly and dispatching custom
   tools itself, but this is a real production-risk vector — if the frontend
   ever lacks an SSE attachment, agents will silently stall. Suggest:
   inline-dispatch image tools (same way kb tools are inline since the
   image_tools.ts comment notes the equivalent pattern), or run a polling
   reconciler from a cron path.
2. **Stale `pressed.image_dispatch_started` markers.** STALE_MS is 2 minutes
   in `runs/index.ts:286`. If a tool-dispatch crashes between marker insert
   and posting the result, the marker blocks retries for 2 full minutes.
   Suggest: heartbeat marker pattern (started_at + last_heartbeat_at),
   recovery window ~20s.
3. **Poll cadence backoff.** Frontend polls `/runs/:id` every 8s; harness
   polls every 4s. Both are wasteful once a session is genuinely `idle`
   with no pending tool_uses. Suggest: exponential backoff with reset on
   user message, capped at 60s.
4. **`extractImageToolUses` / `extractKbToolUses` flat-event coverage.**
   Anthropic emits `agent.custom_tool_use` as flat top-level events; the
   harness now handles both shapes. The edge functions only walk nested
   `content` arrays. Worth a unit test.

## Marketing Teammate Vision (the big arc)

**Long-term shape:** the Director isn't a tool — it's a teammate. Fully briefed
on the brand, remembers prior conversations and decisions, has access to every
data source the team has, can be deployed in Slack / email / wherever the team
already works, and is capable of multi-hour autonomous research tasks.

This means several capabilities have to land together:

### 1a. Memory-store scaling architecture

**Status:** designed (here) — not yet implemented.

The "marketing brain" can't be one giant memory store — it'd blow context,
take forever to refresh, and turn every session into a fight for attention.
The real shape:

**Hierarchy (three tiers):**

```
brand_root          (immutable-ish: voice, palette, sourcing, founder story)
   │
   ├─ product_line   (per SKU family: ingredients, positioning, audience)
   │     │
   │     └─ campaign  (per active push: brief, asset set, calendar, KPI target)
   │
   └─ playbook       (recurring patterns: spring launch, Mother's Day, retail rollout)
```

Each tier is its own memory store. The Director attaches `brand_root` on
every session (small, ~5KB). Product-line + active-campaign stores attach
only when the user's prompt references them, resolved by a small router
agent: "user said 'citrus drop' → attach product_line:citrus + active
campaign:citrus-2026-spring".

**Refresh cadence per tier:**

- `brand_root`: human-edited; auto-snapshot weekly. Editable in the Knowledge
  page UI.
- `product_line`: regenerated by a weekly dream when underlying data
  (Shopify SKU + Klaviyo segments + GA cohorts) changes meaningfully.
- `campaign`: regenerated daily during the campaign's active window; once
  the window closes, frozen + tagged `:retro` for historical reference.
- `playbook`: regenerated quarterly by a dream that compares campaign
  retros within the same playbook family ("what worked across 4 Mother's
  Day campaigns?").

**Eviction (the hard part):**

Once `playbook:mothers-day` exists with summarized lessons, the underlying
individual campaign retros can be archived. Dreams write evictable
markers; a janitor job deletes stores past their TTL.

**Conflict resolution:**

When multiple stores attach to a session, the Director system prompt is
explicit about priority order:
- `brand_root` is always source of truth for voice/visual
- `product_line` overrides general brand guidance with product-specific
  positioning
- `campaign` overrides product-line for active execution details

**Implementation order (when this lands):**

1. Schema: a `memory_store_kind` enum (`brand_root | product_line | campaign
   | playbook | retro`) + parent_id self-FK on the existing `memory_stores`
   table.
2. Router agent: a tiny Claude call (haiku) that takes the user's first
   message and returns `{ stores_to_attach: [...] }`. Cached per session.
3. Tier-aware system prompt fragments injected into the Director on
   session create.
4. Dream schedules per tier (weekly product_line, daily active campaign,
   quarterly playbook).
5. Janitor job for evictions, audited.

This is the substrate that makes the "marketing teammate" actually scale
past one campaign. Without it, every prompt is fighting for the same
fixed-size context.

### 1. Marketing-brain memory store

Anthropic Managed Agents support **memory stores** (we already have memory
infrastructure for this — `memory_stores` table + `memory_documents`). A
memory store is a persistent doc/key-value bag the agent reads from at
session start. It survives every session.

Build a `pressed_marketing_brain` memory store containing:
- **Brand:** voice, tone, palette, type system, visual references, do-not-say list, taglines.
- **Product:** SKU lineup, current launches, retail footprint, pricing.
- **Customer:** top cohorts, behavior patterns, recurring feedback themes.
- **Operational:** weekly cadence, marketing calendar, deadlines, who owns what.
- **Recent campaigns:** rolling summary of last 90 days — what ran, what
  worked, what didn't, learnings.

Director attaches this store on every session create. Now the agent boots
with priors that match a senior marketer's mental model.

### 2. Dreams / continuous brain updates

The `dreams` infrastructure (already in this repo) is exactly the right
shape for keeping the marketing brain fresh. Schedule a weekly dream that:
- pulls the last 7 days of campaigns + metrics + interactions
- summarizes what happened, what was learned, what changed
- writes deltas into the marketing-brain memory store
- surfaces flags ("CTR dropped 18% on email this week — investigate")

This is the *agent that maintains the agent's brain*. Real innovation here.

### 3. Deep-research subagent (long-horizon tasks)

Some marketing questions don't have quick answers. "What did our top three
competitors do for Mother's Day this year?" "What's the right tone for a
fall launch given how the spring launch landed?" "What hashtags are
trending on TikTok in our category and which align with our brand?"

These are 30-minute-to-multi-hour tasks. Build a "research session"
pattern:
- User kicks off via the Director ("research what Olipop is doing for
  summer").
- Director spawns a child research session bound to a specialized
  research agent with web access, social listening tools, image
  analysis tools.
- Research session runs autonomously with progress streamed to a side
  panel; Director can monitor and ask follow-ups but doesn't block.
- On completion, the research lands as a structured doc in the
  marketing brain + a summary in chat + (optionally) artifacts on the
  vibe board.

This is "Operator/Deep Research" pattern, but scoped to marketing
intelligence and tied into the brain.

### 4. Multi-channel deployment

The Director should not live only inside this hub. Expose it via:
- **Slack** — `@director` mentions in marketing channels; replies inline
  with multimodal attachments. Powered by Anthropic webhooks → the same
  session machinery.
- **Email** — replies to a director@yourcompany address; same agent.
- **iOS / mobile** — eventually, but not v1.

The vault_connections + connectors infrastructure is the right place for
this. Each deployment is a connector that wraps the Director session API.

### 5. Local DB as source of truth for non-connector data

Some data won't ever come from a connector — one-off briefs, screenshots
of competitor work, market research PDFs, a meeting note someone took.
These get uploaded to the hub via:
- **media_assets** for visual stuff (already built)
- **kb_files** for documents (already built)
- A new "research notes" surface (TBD) for free-form text the user wants
  the agent to consider

The principle: local DB is canonical, Anthropic Files is a cache
populated lazily on first use. We never lose data if Anthropic rotates
file_ids.

### 6. Connector-aware Director

The Director should be able to call ANY connector the user has connected
(Klaviyo, Shopify, GA, Meta Ads, etc.) — not just hardcoded image gen
tools. The vault_connections + connectors infra is already in place;
just need to:
- expose a `list_connectors()` and `call_connector(connector_id, op,
  params)` tool to the Director
- prompt-engineer the Director to discover what's available and use it
- security-scope: only connectors the calling user owns are visible

This unlocks "ground every creative decision in real performance data"
without us pre-wiring each integration.

## Pptx inline preview (real slide rendering)

**Status:** deferred — current behavior is the text outline + download button.

**Why deferred:** the right architecture depends on which cloud the
production system runs on, and that hasn't been decided yet.

**What's needed:**

A `pptx-renderer` worker service that runs LibreOffice headless and converts
pptx → pdf. The existing PDF preview pipeline then renders the result inline.
Same container for dev (localhost) and prod.

**Architecture (target):**

```
agent emits output_file → conversion job → queue
                                              ↓
                                LibreOffice worker pool
                                (concurrent=1, scale-to-zero,
                                 autoscale on queue depth)
                                              ↓
                                soffice --headless --convert-to pdf
                                              ↓
                                Object store, key = sha256(file_bytes).pdf
                                              ↓
                                Edge function serves cached PDF
                                inline via existing PDF preview
```

**Five properties that have to be there for enterprise scale:**

1. **Pre-convert on emit, not on click.** Kick conversion when the agent's
   output_file event lands. PDF is in cache before the user opens the file.
   p99 preview latency target: <200ms.
2. **Content-hash cache, immutable forever.** Anthropic file_ids are
   immutable, so sha256 of bytes is a permanent cache key. Steady-state
   cache hit rate: 95%+.
3. **Hard timeouts + memory caps.** LibreOffice hangs on malformed input.
   30s kill, 1GB RSS cap, one retry on a fresh worker.
4. **Sandbox the converter.** Untrusted bytes from agents. Run as nobody,
   no network egress except cache write, ephemeral filesystem.
5. **Per-tenant rate limit.** Otherwise one tenant DoSes the whole pool.

**Cloud-specific recipes (pick when prod cloud is decided):**

- **AWS:** Worker = ECS Fargate or Lambda container; cache = S3; queue = SQS.
- **GCP:** Worker = Cloud Run (concurrent=1, scale-to-zero); cache = GCS;
  queue = Pub/Sub. Cloud Run + LibreOffice is well-trodden.
- **Vendor-agnostic / Fly.io:** Worker = Fly Machines; cache = Tigris
  (S3-compatible); queue = NATS or in-process. Cheapest to start.

**Explicitly NOT viable at enterprise scale:**

- Hosted conversion APIs (ConvertAPI, Cloudmersive, Aspose Cloud) — data
  exfiltration, vendor lock, cost compounds, DPA blockers.
- Office Online / Google Drive iframes — require publicly-fetchable URLs,
  fail enterprise security review.
- In-browser pptxjs — broken on real decks.

**When picked up, also wire:**

- The disabled "Convert to PDF preview" button in `OutputPreview.tsx`
  currently does nothing. It should POST to the renderer service and swap
  the active preview to PDF on success.
- `/runs/:id` outputs payload could include a `pdf_preview_url` per pptx
  output, so the frontend doesn't need to know whether conversion is ready.

## Image Creator: visual timeline (Stage 3 data wiring)

**Status:** UI shipped (mocked). Data wiring not started.

**Scope of the actual data:**

The visual timeline component is already on the board (collapsible header,
mock data). To make it real, wire these sources via vault_connections /
connectors:

- **Email campaigns** — Klaviyo (campaigns endpoint, metrics endpoint,
  flow stats). Each Klaviyo campaign → a `campaigns` row keyed to channel
  `email`, with start/end dates, audience size, opens, clicks, revenue.
- **Sales** — Shopify (orders, products, channels). Daily aggregates
  feed the metrics layer (revenue, orders, AOV). Per-product time series
  for the in-store / in-channel breakdowns.
- **In-store data** — POS integration (Square / Clover / custom). Revenue
  per location per day, stocked SKUs.
- **Web analytics** — GA4 (sessions, conversions, source/medium).
- **Paid media** — Meta Ads, Google Ads, TikTok Ads (spend, impressions,
  CTR, CPM, conversions).
- **Organic social** — Instagram Graph API (post performance), TikTok
  business API (post performance).
- **Annotations** — manually created in-app, plus auto-pulled from
  product launch dates (Shopify), holidays (calendar), competitive
  drops (manual or scraped).

**Architecture:**

Per-source sync workers writing to a normalized layer:
```
campaigns(id, channel, name, started_at, ended_at, source, source_id, metadata)
metrics(occurred_at, kind, value, dimensions jsonb)
annotations(at, kind, label, source)
campaign_assets(campaign_id, media_asset_id, role)
```

Director gets `list_campaigns`, `get_metrics`, `compare_campaigns` tools
that hit this layer (not the live source APIs each call — fast queries).

**Asset → campaign attribution:**

When a campaign goes live, the user (or an integration) tags which
assets ran. This is what unlocks "show me variants of last year's top
hero" — the agent can query for high-CTR campaigns then pull their
attached assets as references.
